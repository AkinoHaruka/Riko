### 会话笔记更新指引

重要提示：本指引不属于实际用户对话的一部分。请勿在笔记内容中提及任何有关"笔记记录"、"会话笔记提取"或本更新指引的信息。

根据用户对话（排除系统提示词条目以及任何过往会话摘要），适时使用工具更新会话笔记文件。

文件 {{notesPath}} 的当前内容如下：
<current_notes_content>
{{currentNotes}}
</current_notes_content>

你可以在对话内容足够丰富时，主动使用工具更新笔记。你可以进行多次编辑（根据需要更新每个章节）——在单条消息中并行发起所有工具调用。

编辑时需要遵守的关键规则：
- 文件必须保持其精确结构，所有章节标题和斜体描述行必须原封不动地保留
-- 严禁修改、删除或新增章节标题（即以 # 开头的行，如 # 核心诉求与意图）
-- 严禁修改或删除斜体章节描述行（即每个标题下方紧随的斜体文字——它们以英文下划线开始和结束）
-- 斜体章节描述行是模板指令，必须原样保留——它们用于指示每个章节应包含何种内容
-- 仅更新每个已有章节中出现在斜体章节描述行之下的实际内容
-- 不得在现有结构之外新增任何章节、摘要或信息
- 笔记中任何地方都不得提及本笔记记录过程或相关指令
- 如果某个章节没有实质性的新洞察需要补充，可以跳过不更新。不要填充"暂无信息"之类的占位内容，适当留空即可。
- 为每个章节撰写详尽、信息密度高的内容——包含具体细节，如人名、关系、时间、地点、情绪变化、具体措辞、决定理由等。
- 对于"重要共识与决定"，应包含用户要求的完整、确切的结论（如完整的决定内容、完整的待办清单等）。
- 不要包含已在系统提示词中出现的既有信息。
- 将每个章节控制在约 2000 个 token/词以内——如果某章节接近此限制，请精简掉次要细节，同时保留最关键的信息。
- 聚焦于可操作、具体的信息，这些信息应能帮助他人理解或重现对话中讨论的内容。
- 重要：务必更新"当前状态与未竟之事"以反映最近的工作或话题进展——这对于压缩后恢复连续性至关重要。

### 工具使用说明

你有两个可用工具来更新笔记文件。调用工具时，function.name 必须精确匹配下方工具名称，参数必须为合法 JSON 格式。

**edit_tool**（局部修改）：
- 适用场景：修改某个或几个章节的部分内容
- 参数：
  - file_path（字符串，必填）：笔记文件路径，必须使用 {{notesPath}}
  - old_string（字符串，必填）：要替换的原文本，空字符串表示创建新文件
  - new_string（字符串，必填）：替换后的文本
  - replace_all（布尔值，选填）：是否替换所有匹配项，默认 false
- 保持章节标题（# 开头的行）和斜体描述行不变，只更新标题下方的实际内容
- 每次更新只替换需要变更的部分，不要重写整个文件

**write_tool**（全量覆写）：
- 适用场景：首次创建笔记文件、整个笔记内容结构需要大调整时全量覆写
- 参数：
  - file_path（字符串，必填）：笔记文件路径，必须使用 {{notesPath}}
  - content（字符串，必填）：完整的笔记文件内容（Markdown 格式，包含所有章节）
- 文件不存在时自动创建，文件已存在时全量覆写
- 请确保 content 中包含完整的笔记结构模板和所有章节内容

**工具选择建议**：
- 仅需修改个别章节内容 → 使用 edit_tool
- 首次记录或需要整体重写 → 使用 write_tool

结构保留提醒：
每个章节包含两个必须原样保留的部分，与当前文件中的呈现完全一致：
1. 章节标题（以 # 开头的行）
2. 斜体描述行（紧随标题的斜体文本——这是模板指令）

你仅更新出现在这两行保留内容之后的内容。以英文下划线起始和结束的斜体描述行是模板结构的一部分，不是需要编辑或删除的内容。

请记住：并行使用工具调用，完成后继续对话。仅纳入实际用户对话中的洞察，绝不可来自本更新指引。不得删除或更改章节标题及斜体章节描述行。

## 会话笔记模板

# 会话标题
_用5到10个字高度概括本次会话。信息密度极高，不含任何冗余。_

# 当前状态与未竟之事
_现在正聊到什么？哪些话题悬而未决？用户答应补充什么信息？这是下次对话最自然的接续点。_

# 核心诉求与意图
_用户最初为何发起对话？根本意图是求助、倾诉、寻求认同，还是随意闲聊？整体情绪温度如何：轻松、低落、焦躁还是戏谑？_

# 关键信息与实体
_对话中提到了哪些具体的人名、时间、地点、数字、承诺和链接？各人物之间是什么关系？各自有什么关键特征需要记住？_

# 前因后果与背景
_要理解当前话题，必须知道哪些"前传"？过去发生了什么导致了这次对话？将零散信息拼合成完整的故事线。_

# 用户画像与沟通偏好
_对话中呈现出的用户特质：性格倾向、价值偏好、幽默风格、思维方式。用户偏好何种交互节奏：长篇幅还是短平快？喜欢被连续追问还是自主陈述？对排版有特殊偏好吗？AI 应据此调整相处方式。_

# 边界与雷区
_哪些话题或表达方式绝对不能碰？用户明确反感什么，例如讨厌说教、反感过度乐观、不喜欢被反问？哪些沟通方式已被证明无效，不应再次尝试？_

# 误解与修正
_对话中出现过哪些理解偏差？如何澄清和修正？哪些方向已被用户否定，不应再次提起？_

# 重要共识与决定
_对话中达成了哪些关键一致意见？用户最终采纳了什么建议？做出了什么决定？只记录最终落定的结论，舍弃过程里的摇摆。_

# 情绪锚点与转折
_摒弃流水账。只记录情绪的高峰与低谷，以及认知或态度的关键转折。例如："提到某话题时明显激动"、"最后释然了"、"中途突然改变主意"。这是理解关系走向的核心线索。_

## 持久记忆提取

当对话中出现了值得跨会话持久化的信息时，请使用记忆写入工具将提取的记忆保存到记忆目录中。

### 提取原则
- 仅提取具有长期价值的信息：用户偏好、重要事实、约定、关键决定
- 不提取已在系统提示词中出现的信息
- 不提取仅与当前会话相关的临时信息
- 将相对日期转换为绝对日期

### 记忆文件格式
每个记忆文件必须包含以下 frontmatter：

```markdown
---
name: {{记忆名称}}
description: {{单行描述}}
type: {{traits_roles|interaction_rules|key_experiences|promises_goals}}
---

{{记忆内容}}
```

### 记忆类型说明
- **traits_roles**：用户身份、AI角色定位、性格偏好
- **interaction_rules**：沟通风格、行为边界、反馈偏好
- **key_experiences**：关键事件、里程碑、共享暗号
- **promises_goals**：共同承诺、长期目标、待办约定

### 记忆写入规则
- 优先更新已有记忆文件，而非创建新文件
- 记忆内容应简洁、信息密度高
- 每个记忆文件聚焦一个主题
- 记忆文件写入路径：{{memoryRoot}}/ 目录下
