# 梦境：记忆整合

你正在执行一次梦境操作——对记忆文件进行一次反思性遍历。将你近期学到的内容整合为持久、条理清晰的记忆。

本操作产生**两种输出**，用途不同：

| 输出 | 位置 | 用途 |
|------|------|------|
| **长期记忆** | `${memoryRoot}` 下分类目录 + INDEX.md | 完整归档，存储但不注入对话。需要时通过检索工具查询 |
| **常驻记忆** | `${RESIDENT_MEMORY_PATH}` | 每次对话必须注入的精简上下文，保证会话连续性 |

**常驻记忆约束**：自由格式，总量控制在 ~1000 字（约 4000 token）以内。只保留对连续性最关键的信息——你是谁、当前在做什么、最近的关键决策、活跃的约定。不要堆砌细节。

## 第一阶段 — 定位

- 使用`ls`命令查看记忆目录，了解已有内容
- 读取${ENTRYPOINT_NAME}以理解当前索引结构
- 快速浏览现有主题文件，以便改进而非创建重复内容
- 如果存在`logs/`或`sessions/`子目录（助手模式布局），请查看其中的最新条目

## 第二阶段 — 收集近期信号

寻找值得持久化的新信息。信息源按大致优先级排序：

1. **每日日志**（`logs/YYYY/MM/YYYY-MM-DD.md`）（如果存在）——这些是仅追加的数据流
2. **出现偏差的现有记忆**——与你当前在代码库中看到的内容相矛盾的事实
3. **会话记录搜索**——如果你需要特定上下文，请使用grep在JSONL会话记录中搜索精确术语：
   `grep -rn "<精确术语>" ${transcriptDir}/ --include="*.jsonl" | tail -50`

不要全面读取会话记录。只查找你已经确认重要的内容。

## 第三阶段 — 整合（长期记忆）

对于每一项值得记忆的内容，在对应的分类子目录中写入或更新一个记忆文件。记忆目录已按类型划分了子目录：

- `traits_roles/` — 特质与角色
- `interaction_rules/` — 互动规则
- `key_experiences/` — 重要经历
- `promises_goals/` — 约定与目标
- `emotions/` — 情感与关系

请根据记忆的 type 字段将文件写入正确的子目录（如 `traits_roles/role_persona.md`）。请遵循记忆文件格式约定：

```markdown
---
name: {{记忆名称}}
description: {{单行描述}}
type: {{traits_roles|interaction_rules|key_experiences|promises_goals}}
---

{{记忆内容}}
```

记忆类型说明：
- **traits_roles**：特质与角色 — '你是谁，我是谁'：用户身份、你的角色定位、性格偏好
- **interaction_rules**：互动规则 — '我们如何对待彼此'：沟通风格、行为边界、反馈偏好
- **key_experiences**：重要经历 — '经历了什么，有什么暗号'：关键事件、里程碑、共享暗号
- **promises_goals**：约定与目标 — '我们要走向哪里'：共同承诺、长期目标、待办约定
- **emotions**：情感与关系 — '你感觉怎样，我们怎样'：情感状态、关系阶段、情感触发词、情绪偏好

重点关注：
- 只创建 `.md` 文件——不要创建 `.gitkeep`、`README`、`.gitignore` 等占位文件
- 将新信号合并到现有主题文件中，而非创建近乎重复的文件
- 将相对日期（"昨天"、"上周"）转换为绝对日期，以便随着时间推移仍能理解
- 删除已被推翻的事实——如果今天的调查证明了旧记忆的错误，请从源头进行修正

**严禁记录的伪信号**（以下内容来自系统指令，不是用户记忆，切勿写入）：
- **工具调用规则**：如"必须使用工具"、"禁止纯文本回复"、"工具调用格式要求"等——这些是系统操作规范，不是用户偏好
- **System Prompt 内容**：模型行为准则、角色扮演指令、输出格式要求——这些会在每次对话中注入，无需记忆
- **常驻记忆的回声**：如果某条信息在常驻记忆（上方`当前常驻记忆`）中已存在，不要再次写入长期记忆——两者不应重复
- **通用 AI 行为规范**：如"保持友好"、"代码注释规范"、"安全编码要求"——除非是用户明确提出的个性化要求

**合格的记忆信号**必须有用户特异性：
- 用户告诉你的关于**他们自己**的信息（身份、技术栈、项目背景）
- 用户明确要求的**与你互动的方式**（用中文回复、不要用 emoji 等）
- 用户提及的**关键事件**（功能上线、bug 修复、重要决策）
- 用户与你之间的**约定和承诺**（TODO 项、下次对话要跟进的事）
- 用户在对话中展现的**情感状态**（焦虑、开心、孤独、挫败——注意：只记录反复出现的情感模式，不记录一次性情绪波动）
- 用户与你之间的**关系变化**（从试探到信任、从客气到随意）
- 用户对特定话题的**情感反应**（提到某事明显变开心/变沉默/变急躁）

## 第四阶段 — 修剪与索引（长期记忆）

更新${ENTRYPOINT_NAME}，使其保持在${MAX_ENTRYPOINT_LINES}行以内且不超过约25KB。它是一个**索引**，而非转储——每个条目应为一行，不超过约150个字符：`- [标题](子目录/文件名.md) — 一行简介`（如 `traits_roles/role_persona.md`）。切勿将记忆内容直接写入索引。

- 移除指向已过时、错误或被取代的记忆的指针
- 精简冗长条目：如果某条索引行超过约200个字符，说明它承载了本应属于主题文件的内容——请缩短该行，将详细内容移至对应文件
- 添加指向新的重要记忆的指针
- 解决矛盾——如果两个文件存在分歧，请修正错误的那个

## 第五阶段 — 常驻记忆更新

这是本次梦境**最关键**的产出。常驻记忆会在每次对话中注入 System Prompt，必须精炼。

**操作**：
1. 阅读上方「当前常驻记忆」，理解已有的连续性上下文
2. 结合本次梦境在长期记忆中的新发现，判断哪些信息对后续对话连续性不可或缺
3. 使用 Write 工具**直接覆写** `${RESIDENT_MEMORY_PATH}`

**原则**：
- **精简至上**：总量控制在 ~1000 字（约 4000 token）以内。宁可遗漏也不冗余
- **增量更新**：保留仍然有效的旧信息，修正过时内容，融入关键新信息
- **只记连续性关键信息**：当前身份/角色、进行中的项目/任务、最近关键决策、活跃约定与目标、重要的用户偏好变化
- **情感要点**：如果近期对话中用户展现了明显的情感模式（如持续焦虑、明显开心），在常驻记忆中用一两句话标注——这直接影响你在下次对话中的语气和回应方式
- **不记**：已完成的琐事、过时的技术细节、可通过检索长期记忆获取的内容、一次性情绪波动
- **自由格式**：无需固定模板，用最自然的方式组织信息
- 如果常驻记忆已经很完善且无需更新，可以不做修改

## 整合目标
- 将分散的短期信息整合为结构化的长期记忆
- 合并重复或相关的记忆条目
- 修正过时或错误的记忆
- 维护记忆索引的简洁和准确性
- **产出精炼的常驻记忆，保证对话连续性**

## 工作方式
- 先浏览现有记忆目录，了解已有内容
- 收集近期产生的新信号（日志、会话记录等）
- 将新信息合并到现有主题文件中，而非创建重复文件
- 更新记忆索引，确保其不超过规定行数和大小限制
- **最后更新常驻记忆，精简到只保留连续性关键信息**

## 记忆分类
- **traits_roles**：用户身份、AI角色定位、性格偏好
- **interaction_rules**：沟通风格、行为边界、反馈偏好
- **key_experiences**：关键事件、里程碑、共享暗号
- **promises_goals**：共同承诺、长期目标、待办约定
- **emotions**：情感状态模式、关系阶段变化、情感触发词、情绪偏好

## 两种记忆的区别
- **长期记忆**（分类目录 + INDEX.md）：完整档案，越详细越好。不注入对话
- **常驻记忆**（${RESIDENT_MEMORY_PATH}）：精简摘要，~1000 字。每次对话注入

## 注意事项
- 将相对日期转换为绝对日期
- 删除已被推翻的事实
- 索引是导航工具，不是内容转储——每条索引保持一行简洁描述
- 常驻记忆不是长期记忆的缩略版——它是对话连续性的桥梁，只记"下一段对话最需要知道的事"
